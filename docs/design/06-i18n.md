# 06 — Internationalization

## 6.1 Goals

| Goal                                                                      | Acceptance criterion                                                                                 |
| ------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------- |
| Locale data is **lazily loaded**                                          | A user that imports `@kumiki/components/combobox` and never uses Calendar pulls in zero locale data. |
| Each locale package is **≤ 1 KB gzipped**                                 | `size-limit` budget per `@kumiki/locale/<lang>` entry.                                               |
| **RTL keyboard** semantics are correct                                    | `ArrowLeft` reverses meaning in `dir="rtl"`; tests cover Tabs, RadioGroup, Combobox, Slider.         |
| **Non-Gregorian calendars** are first-class (Phase 2 Calendar/DatePicker) | Gregorian, Islamic, Buddhist, Japanese, Hebrew calendars are independent dynamic imports.            |
| **Numbering systems** are user-controlled                                 | Latin (default), Arabic-Indic, Persian, Thai, Devanagari supported via `Intl.NumberFormat`.          |

## 6.2 v1.0 locale set

Ten languages ship as locale data at v1.0:

| Code      | Language            | Script | Direction | Why included                          |
| --------- | ------------------- | ------ | --------- | ------------------------------------- |
| `en`      | English             | Latin  | LTR       | Default; ~25% of web users            |
| `ja`      | Japanese            | CJK    | LTR       | Primary author's locale; CJK coverage |
| `zh-Hans` | Simplified Chinese  | CJK    | LTR       | Largest CJK reading population        |
| `zh-Hant` | Traditional Chinese | CJK    | LTR       | Distinct from Hans for content        |
| `ko`      | Korean              | Hangul | LTR       | CJK coverage completion               |
| `es`      | Spanish             | Latin  | LTR       | Largest Romance population            |
| `fr`      | French              | Latin  | LTR       | Government / EU baseline              |
| `de`      | German              | Latin  | LTR       | DACH enterprise baseline              |
| `ar`      | Arabic              | Arabic | **RTL**   | RTL coverage + non-Latin numerals     |
| `he`      | Hebrew              | Hebrew | **RTL**   | RTL coverage + distinct script        |

Phase 2 expansion: `pt`, `it`, `nl`, `tr`, `pl`, `ru`, `id`, `vi`, `th`, `fa`. (Brings to 20.) Final 30+ depends on community contributions.

## 6.3 Locale data structure

Each `@kumiki/locale/<lang>` exports a typed messages object:

```ts
// @kumiki/locale/ja — sketch
export const messages = {
  combobox: {
    listboxLabel: '候補一覧',
    countResults: (n: number) => (n === 0 ? '候補なし' : `${n} 件の候補`),
    cleared: '入力をクリアしました',
  },
  dialog: {
    closeLabel: '閉じる',
  },
  // ... per-component, intentionally narrow
} as const;

export type Messages = typeof messages;
```

Every other locale exports the same shape (TypeScript enforces this via `Messages` type imported from `@kumiki/types`).

**ICU MessageFormat?** Not at v1.0. We use plain template functions. ICU is overkill for the strings we have, and it would force us to ship an ICU runtime parser — the parser alone is bigger than our entire `@kumiki/locale/*` per-language budget. If a Phase 2 component needs full ICU semantics (plural forms beyond `n === 0`/`n === 1`/`n > 1`, gender variants), we revisit.

## 6.4 The provider model

A consumer wraps their app once:

```svelte
<!-- src/routes/+layout.svelte -->
<script lang="ts">
  import { LocaleProvider } from '@kumiki/components/locale-provider';
  import { messages, direction } from '@kumiki/locale/ja';

  let { children } = $props();
</script>

<LocaleProvider.Root locale="ja" {messages} dir={direction}>
  {@render children()}
</LocaleProvider.Root>
```

`<LocaleProvider.Root>` (Layer 4 — the Svelte component lives in
`@kumiki/components` because it touches the framework runtime) sets a
Svelte context for:

- `locale: string` — BCP-47 tag, used by `Intl.NumberFormat`, `Intl.Collator`, `Intl.DateTimeFormat`.
- `messages: Messages` — per-locale message bundle.
- `dir: 'ltr' | 'rtl' | 'auto'` — `'auto'` infers from `locale` via `inferDirection()` from `@kumiki/primitives/locale`.

The pure-TS pieces (the symbol key, the `Direction` type, and the
`inferDirection()` helper) live at `@kumiki/primitives/locale` so any
non-Svelte tooling can read or set the same context.

Components consume via `useLocale()`:

```ts
// Inside a Layer 4 component
import { useLocale } from '@kumiki/components/locale-provider';

const { messages, dir, locale } = useLocale();
```

`tryUseLocale()` returns `undefined` instead of throwing when no
provider is in scope, for components that should gracefully fall back
to default English strings.

## 6.5 Dynamic locale loading

Vite's static analysis of dynamic-import template literals produces one chunk per locale. We rely on it:

```ts
type LocaleCode = 'en' | 'ja' | 'zh-Hans' | 'zh-Hant' | 'ko' | 'es' | 'fr' | 'de' | 'ar' | 'he';

async function loadLocale(code: LocaleCode) {
  const { messages } = await import(`@kumiki/locale/${code}`);
  return messages;
}
```

The Phase 0c `apps/docs/sizes/locale-tree-shake.test.ts` verifies that each chunk is ≤ 1 KB gzipped and that no unrequested locale is in the bundle. The test fails CI if a regression bundles all locales together.

## 6.6 RTL semantics

### Direction propagation

`dir="auto"` on `LocaleProvider` infers from the locale (`ar` and `he` → `rtl`; everything else in v1.0 → `ltr`). Explicit override via `dir="rtl"` always wins.

The provider sets `data-direction` on the wrapping element and exposes it via context. Components emit `data-direction` on their root for CSS hooks.

### Keyboard logic

Keyboard interactions that depend on physical direction (e.g., Tabs `ArrowRight` to move forward) are inverted in `rtl`:

```ts
// Inside the Tabs machine
function nextDirection(
  dir: 'ltr' | 'rtl',
  key: 'ArrowLeft' | 'ArrowRight',
): 'forward' | 'backward' {
  if (dir === 'ltr') return key === 'ArrowRight' ? 'forward' : 'backward';
  return key === 'ArrowRight' ? 'backward' : 'forward';
}
```

The machine takes `dir` as part of its context (from the locale provider) and uses `nextDirection` in its transition logic. RTL behavior is a machine concern, not a controller concern.

### Test matrix

Every keyboard-driven component runs the keyboard test suite in both `ltr` and `rtl`. Adding a new keyboard interaction requires updating both directions in the YAML.

## 6.7 Numbering systems

`Intl.NumberFormat` handles the conversion automatically when given a locale with a numbering system extension:

```ts
new Intl.NumberFormat('ar-EG', { numberingSystem: 'arab' }).format(1234); // "١٬٢٣٤"
new Intl.NumberFormat('fa-IR', { numberingSystem: 'arabext' }).format(1234); // "۱٬۲۳۴"
new Intl.NumberFormat('th-TH', { numberingSystem: 'thai' }).format(1234); // "๑,๒๓๔"
```

We wrap this in `@kumiki/primitives/locale.formatNumber(n, options)`, which reads the current locale from context and applies its numbering-system extension if any. Number-input components (Phase 2 NumberField) accept localized digit strings as input via a parser primitive.

## 6.8 Calendars (Phase 2)

Phase 2 ships `@kumiki/components/calendar` and `@kumiki/components/date-picker`. Both consume `@internationalized/date` from React Aria — it is **framework-agnostic** despite the name and ships as a peer dependency:

```json
"peerDependencies": {
  "@internationalized/date": "^3.12.0"
}
```

`@internationalized/date` supports 13 calendars (Gregorian, Buddhist, Ethiopic, Hebrew, Indian, Islamic — 3 variants —, Japanese, Persian, ROC, etc.). At ~8 KB gzipped _full_, we tree-shake by importing only the calendars we use; a Gregorian-only subset is closer to 2.8 KB ([@internationalized/date](https://react-spectrum.adobe.com/internationalized/date/)).

For Phase 2 we will provide:

```ts
import { GregorianCalendar } from '@internationalized/date';
// ...
import {
  IslamicCivilCalendar,
  IslamicTabularCalendar,
  IslamicUmalquraCalendar,
} from '@internationalized/date';
import { BuddhistCalendar } from '@internationalized/date';
import { JapaneseCalendar } from '@internationalized/date';
import { HebrewCalendar } from '@internationalized/date';
```

Each is its own import path, so `pnpm add @kumiki/components` + `import { JapaneseCalendar } from '@internationalized/date'` doesn't pull in Hebrew or Buddhist.

**TC39 Temporal status (May 2026):** Stage 4. Firefox 139 + Chrome/Edge 144 + V8 ship. Safari and Node lag. We do **not** target Temporal natively at v1.0 because most users would still need a polyfill (~20 KB). When Safari and Node ship, we add a Temporal-based date primitive in parallel. Adobe has signalled the same migration path for `@internationalized/date`.

## 6.9 Locale data — what's in the v1.0 strings

Per locale, we ship strings for:

- Combobox: listbox label, "no results", count of results, clear input.
- Dialog: "Close" label.
- Tabs: tablist label fallback when none provided.
- FormField: required indicator, generic validation messages (required, type mismatch).
- Tooltip: nothing (tooltips are content-only).
- Toggle / Switch / Checkbox / RadioGroup / Select: standard "ON / OFF / SELECTED / NOT SELECTED" patterns are inherent to ARIA roles; no library strings needed.

This is a **deliberately small set**. Per-component messages are kept minimal so the per-locale gzip budget holds.

## 6.10 Alternatives we considered

| Alternative                                                                                                 | Verdict                                                              |
| ----------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------- |
| **Bundle all locales by default**, like Bits UI                                                             | ❌ blows the size budget; users pay for languages they don't speak   |
| **ICU MessageFormat at v1.0**                                                                               | ❌ runtime parser is ~10 KB; we have ~50 strings total; not worth it |
| **Deep React Aria style** with full string-aware components, locale matchers, fallbacks                     | ⚠️ too much surface for v1.0; revisit in v1.x                        |
| **Defer i18n to user-land entirely** (Bits UI / Melt UI shape)                                              | ❌ this is a competitive differentiator; we can't punt               |
| **Chosen:** plain template messages, lazy-loaded, dir-aware, calendar plug-in via `@internationalized/date` | ✅                                                                   |

- **TBD:** Pluralization beyond zero / one / many. Currently a function returns the right phrase. When a Phase 2 component needs Russian-style six-form plurals, we may adopt a tiny plural-rules helper (~500 B) — or introduce ICU MessageFormat for that single component.
